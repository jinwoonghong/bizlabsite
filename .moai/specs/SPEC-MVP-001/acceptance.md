# SPEC-MVP-001 인수 기준

## 1. 모듈 1: 데이터 모델 및 API

### 시나리오 1-1: 논문 등록 API 성공
- **Given**: 서버가 정상 동작 중인 상태
- **When**: `POST /api/papers` 요청이 유효한 데이터(`title`, `authors`, `year`)와 함께 전송됨
- **Then**: 201 상태 코드와 생성된 논문 객체(id, createdAt 포함)가 반환됨

### 시나리오 1-2: 필수 필드 누락 시 등록 실패
- **Given**: 서버가 정상 동작 중인 상태
- **When**: `POST /api/papers` 요청에서 `title` 필드가 누락됨
- **Then**: 400 상태 코드와 `"title은 필수 항목입니다"` 에러 메시지가 반환됨

### 시나리오 1-3: 유효하지 않은 URL 형식
- **Given**: 서버가 정상 동작 중인 상태
- **When**: `POST /api/papers` 요청에서 `link` 필드에 `"not-a-url"` 값이 전송됨
- **Then**: 400 상태 코드와 URL 형식 에러 메시지가 반환됨

### 시나리오 1-4: 논문 목록 조회
- **Given**: 데이터베이스에 3건의 논문이 등록된 상태
- **When**: `GET /api/papers` 요청이 전송됨
- **Then**: 200 상태 코드와 3건의 논문 배열이 반환됨

### 시나리오 1-5: 존재하지 않는 논문 조회
- **Given**: 데이터베이스에 ID 999인 논문이 존재하지 않는 상태
- **When**: `GET /api/papers/999` 요청이 전송됨
- **Then**: 404 상태 코드와 `"논문을 찾을 수 없습니다"` 에러 메시지가 반환됨

### 시나리오 1-6: 논문 수정 API (비밀번호 검증 성공)
- **Given**: ID 1인 논문이 존재하고 `ADMIN_PASSWORD`가 `"admin123"`인 상태
- **When**: `PUT /api/papers/1` 요청이 `password: "admin123"`과 수정 데이터와 함께 전송됨
- **Then**: 200 상태 코드와 갱신된 논문 객체가 반환되며 `updatedAt`이 갱신됨

### 시나리오 1-7: 논문 삭제 API (비밀번호 불일치)
- **Given**: ID 1인 논문이 존재하고 `ADMIN_PASSWORD`가 `"admin123"`인 상태
- **When**: `DELETE /api/papers/1` 요청이 `password: "wrong"`과 함께 전송됨
- **Then**: 401 상태 코드와 `"비밀번호가 올바르지 않습니다"` 에러 메시지가 반환됨

## 2. 모듈 2: 논문 등록/조회

### 시나리오 2-1: 논문 등록 폼 표시
- **Given**: 사용자가 `/papers/new` 페이지에 접속한 상태
- **When**: 페이지가 로드됨
- **Then**: 제목, 저자, 발행연도(필수), 저널명, 링크, 초록(선택) 입력 필드와 등록 버튼이 표시됨

### 시나리오 2-2: 논문 등록 성공 후 리다이렉트
- **Given**: 사용자가 논문 등록 페이지에서 필수 필드를 입력한 상태
- **When**: 등록 버튼을 클릭함
- **Then**: 논문이 데이터베이스에 저장되고 메인 페이지(`/`)로 리다이렉트되며 목록에 새 논문이 표시됨

### 시나리오 2-3: 빈 필수 필드 제출 시 클라이언트 검증
- **Given**: 사용자가 논문 등록 페이지에서 제목을 입력하지 않은 상태
- **When**: 등록 버튼을 클릭함
- **Then**: 제목 필드 아래에 검증 에러 메시지가 표시되고 폼이 제출되지 않음

### 시나리오 2-4: 논문 상세 페이지 표시
- **Given**: ID 1인 논문이 데이터베이스에 존재하는 상태
- **When**: 사용자가 `/papers/1` 페이지에 접속함
- **Then**: 논문의 제목, 저자, 발행연도, 저널명, 링크, 초록, 등록일시가 모두 표시됨

### 시나리오 2-5: 빈 논문 목록 상태
- **Given**: 데이터베이스에 논문이 없는 상태
- **When**: 사용자가 메인 페이지에 접속함
- **Then**: "등록된 논문이 없습니다" 메시지와 논문 등록 링크가 표시됨

### 시나리오 2-6: 원문 링크 표시
- **Given**: 논문에 `link` 값이 `"https://arxiv.org/abs/1234"`인 상태
- **When**: 사용자가 해당 논문 상세 페이지를 조회함
- **Then**: "원문 보기" 링크가 `target="_blank"`, `rel="noopener noreferrer"` 속성과 함께 표시됨

## 3. 모듈 3: 논문 수정/삭제 및 관리자 인증

### 시나리오 3-1: 수정 시 비밀번호 모달 표시
- **Given**: 사용자가 논문 상세 페이지에 있는 상태
- **When**: "수정" 버튼을 클릭함
- **Then**: 관리자 비밀번호 입력 모달이 표시됨

### 시나리오 3-2: 올바른 비밀번호로 수정 페이지 이동
- **Given**: 관리자 비밀번호 모달이 표시된 상태
- **When**: 올바른 비밀번호를 입력하고 확인 버튼을 클릭함
- **Then**: 논문 수정 페이지(`/papers/[id]/edit`)로 이동하며 기존 데이터가 폼에 채워져 있음

### 시나리오 3-3: 잘못된 비밀번호 입력
- **Given**: 관리자 비밀번호 모달이 표시된 상태
- **When**: 잘못된 비밀번호를 입력하고 확인 버튼을 클릭함
- **Then**: 모달 내에 "비밀번호가 올바르지 않습니다" 에러 메시지가 표시되고 페이지 이동 없음

### 시나리오 3-4: 논문 수정 성공
- **Given**: 사용자가 논문 수정 페이지에서 제목을 변경한 상태
- **When**: 저장 버튼을 클릭함
- **Then**: 논문이 갱신되고 상세 페이지로 리다이렉트되며 변경된 제목이 표시됨

### 시나리오 3-5: 논문 삭제 확인 및 성공
- **Given**: 사용자가 논문 상세 페이지에서 "삭제" 버튼을 클릭하고 비밀번호 모달이 표시된 상태
- **When**: 올바른 비밀번호를 입력하고 삭제 확인 버튼을 클릭함
- **Then**: 논문이 삭제되고 메인 페이지로 리다이렉트되며 해당 논문이 목록에서 사라짐

### 시나리오 3-6: 비밀번호 모달 닫기
- **Given**: 관리자 비밀번호 모달이 표시된 상태
- **When**: ESC 키를 누르거나 모달 배경을 클릭함
- **Then**: 모달이 닫히고 이전 상태로 돌아감

## 4. 모듈 4: 검색 및 필터링

### 시나리오 4-1: 키워드 검색
- **Given**: 메인 페이지에 "Deep Learning", "Machine Learning", "Natural Language" 제목의 논문 3건이 표시된 상태
- **When**: 검색창에 "Learning"을 입력함
- **Then**: "Deep Learning"과 "Machine Learning" 논문 2건이 표시됨

### 시나리오 4-2: 연도별 필터링
- **Given**: 2023년과 2024년 논문이 각각 존재하는 상태
- **When**: 연도 필터에서 "2024"를 선택함
- **Then**: 2024년 논문만 표시됨

### 시나리오 4-3: 정렬 변경
- **Given**: 메인 페이지에 논문이 최신순으로 표시된 상태
- **When**: 정렬 옵션을 "제목순"으로 변경함
- **Then**: 논문 목록이 제목 알파벳/가나다 순으로 재정렬됨

### 시나리오 4-4: 검색 결과 없음
- **Given**: 메인 페이지에 논문이 표시된 상태
- **When**: 검색창에 "ZZZZZ_존재하지_않는_키워드"를 입력함
- **Then**: "검색 결과가 없습니다" 메시지와 필터 초기화 버튼이 표시됨

### 시나리오 4-5: URL 쿼리 파라미터 반영
- **Given**: 사용자가 검색어 "AI"와 연도 "2024"로 필터링한 상태
- **When**: 브라우저 주소창의 URL을 확인함
- **Then**: URL에 `?query=AI&year=2024` 쿼리 파라미터가 포함됨

### 시나리오 4-6: 디바운스 적용 검색
- **Given**: 사용자가 검색창에 포커스한 상태
- **When**: "abc"를 빠르게 연속 타이핑함
- **Then**: 타이핑이 멈춘 후 300ms 뒤에 한 번만 검색이 실행됨 (중간 결과 없음)

## 5. 모듈 5: 반응형 UI 및 레이아웃

### 시나리오 5-1: 모바일 레이아웃 (< 768px)
- **Given**: 화면 너비가 375px인 모바일 디바이스
- **When**: 메인 페이지에 접속함
- **Then**: 논문 카드가 단일 열(1 column)로 수직 배치되며, Header와 Footer가 전체 너비를 차지함

### 시나리오 5-2: 태블릿 레이아웃 (768px ~ 1023px)
- **Given**: 화면 너비가 768px인 태블릿 디바이스
- **When**: 메인 페이지에 접속함
- **Then**: 논문 카드가 2열(2 columns) 그리드로 표시됨

### 시나리오 5-3: 데스크탑 레이아웃 (>= 1024px)
- **Given**: 화면 너비가 1280px인 데스크탑 브라우저
- **When**: 메인 페이지에 접속함
- **Then**: 논문 카드가 3열(3 columns) 그리드로 표시됨

### 시나리오 5-4: Header 네비게이션
- **Given**: 사용자가 어떤 페이지에 있든지
- **When**: 페이지를 조회함
- **Then**: Header에 서비스 이름/로고와 "논문 등록" 링크가 표시됨

### 시나리오 5-5: 로딩 상태 표시
- **Given**: 사용자가 메인 페이지에 접속한 상태
- **When**: 논문 목록 데이터를 로드 중임
- **Then**: 로딩 인디케이터(스피너 또는 스켈레톤 UI)가 표시됨

### 시나리오 5-6: 네트워크 오류 시 에러 처리
- **Given**: 사용자가 논문 등록 폼을 작성한 상태
- **When**: 네트워크 오류로 폼 제출이 실패함
- **Then**: 사용자 친화적 에러 메시지가 표시되고 입력한 데이터가 보존됨

## 6. 엣지 케이스

### 시나리오 E-1: 매우 긴 텍스트 입력
- **Given**: 사용자가 논문 등록 페이지에 있는 상태
- **When**: 제목에 500자 이상의 텍스트를 입력함
- **Then**: UI가 깨지지 않고 텍스트가 적절히 줄바꿈 또는 말줄임 처리됨

### 시나리오 E-2: 특수문자 포함 검색
- **Given**: 논문 제목에 특수문자(`<script>`, `'`, `"`)가 포함된 논문이 존재하는 상태
- **When**: 해당 특수문자로 검색함
- **Then**: XSS 공격 없이 안전하게 결과가 표시되며 해당 논문이 검색됨

### 시나리오 E-3: 동시 수정 시도
- **Given**: 두 명의 사용자가 동일한 논문 수정 페이지를 열어놓은 상태
- **When**: 양쪽에서 서로 다른 수정사항을 저장함
- **Then**: 마지막으로 저장한 수정사항이 반영됨 (Last-Write-Wins)

### 시나리오 E-4: 빈 문자열 필드 처리
- **Given**: 사용자가 논문 등록 페이지에 있는 상태
- **When**: 제목에 공백만 입력하고 등록 버튼을 클릭함
- **Then**: 검증 에러가 발생하며 "제목을 입력해주세요" 메시지가 표시됨

### 시나리오 E-5: 연도 범위 검증
- **Given**: 사용자가 논문 등록 페이지에 있는 상태
- **When**: 발행연도에 `1800` 또는 `2100` 등 비현실적 연도를 입력함
- **Then**: 검증 에러가 발생하며 합리적인 연도 범위(1900~현재연도+1) 안내 메시지가 표시됨

## 7. 품질 게이트

### 7.1 테스트 커버리지
- 전체 코드 커버리지: 85% 이상
- API Route 커버리지: 90% 이상
- 컴포넌트 렌더링 테스트: 각 컴포넌트 최소 1개 테스트

### 7.2 타입 안전성
- TypeScript 에러: 0건 (`npx tsc --noEmit` 통과)
- ESLint 에러: 0건 (`npm run lint` 통과)

### 7.3 반응형 검증
- 모바일 (360px, 375px): 1열 레이아웃 확인
- 태블릿 (768px): 2열 레이아웃 확인
- 데스크탑 (1024px, 1280px): 3열 레이아웃 확인

### 7.4 성능 기준
- API 목록 조회 응답 시간: 200ms 이하 (논문 100건 기준)
- 페이지 초기 로드 (LCP): 2.5초 이하
- Lighthouse 성능 점수: 80점 이상

### 7.5 접근성
- 모든 이미지에 alt 텍스트
- 폼 필드에 label 연결
- 키보드 네비게이션 지원 (Tab, Enter, ESC)
- 적절한 색상 대비 (WCAG AA 기준)

### 7.6 보안
- XSS 방지: 사용자 입력 이스케이프 처리 (React 기본 제공)
- ADMIN_PASSWORD 서버 측 전용 (클라이언트 노출 금지)
- API 에러 메시지에 내부 구현 정보 미노출

- [ ] 검색, 필터링, 정렬 기능이 정상 동작
- [ ] 관리자 비밀번호 인증이 서버 측에서 동작
- [ ] TypeScript 컴파일 에러 0건
- [ ] ESLint 에러 0건
- [ ] 테스트 커버리지 85% 이상
- [ ] 로딩/에러 상태가 적절히 처리됨
- [ ] 빈 상태(논문 없음, 검색 결과 없음) UI가 구현됨
- [ ] 모든 폼에 클라이언트 + 서버 측 검증 적용
